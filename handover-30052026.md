## 3. START HERE — what was NOT verifiable on Windows

The Windows machine had **no Android toolchain** and could **not** build
`sovereign-app` itself (it pulls `sovereign-ai` → `llama-cpp-2`, needing
CMake/LLVM/clang). So these are unproven and are your first tasks:

1. **Confirm the whole binary compiles** before the long Android pipeline:
   ```bash
   # mobile feature set: encryption + comms-signal + haptics, NO cuda, NO rocksdb
   cargo check -p sovereign-app --no-default-features --features mobile -j 4
   ```
   This is the real test of the `ea96d75` fix (the `pii_sweep` module only
   compiles when `comms + encryption` are both on — exactly what `mobile`
   enables). If anything else from the device_key→account_key migration was
   missed, it surfaces here.

2. **Run the integration/heavy tests** that needed a toolchain:
   ```bash
   cargo test -p sovereign-app --no-default-features --features mobile -j 2   # incl. account_key_migration tests
   cargo test -p sovereign-comms -j 2
   cargo test -p sovereign-ai --no-default-features -j 2     # sovereign-ai DEFAULTS to cuda — disable it
   ```
   Note `account_key_migration.rs` has tests (e.g.
   `migrates_pii_record_from_device_key_to_account_key`) that only run as part
   of `sovereign-app`.

3. **Build the APK** — full toolchain + steps are in
   `doc/plans/v0.0.5-release-handoff.md`. Short version:
   ```bash
   rustup target add aarch64-linux-android armv7-linux-androideabi i686-linux-android x86_64-linux-android
   # JDK 17, Android SDK + NDK, ANDROID_HOME/NDK_HOME set, cargo tauri-cli ^2
   node scripts/tauri-build-frontend.cjs        # sanity-check the cwd-agnostic frontend resolver
   cargo tauri android build --debug --target aarch64
   ```
   Frontend resolution for Android is overridden by `tauri.android.conf.json`
   (`beforeBuildCommand` → `scripts/tauri-build-frontend.cjs`) because Tauri runs
   the Android beforeBuild from `crates/`, not `crates/sovereign-app/`.
   `gen/android/` is committed and hand-patched (manifest perms:
   ACCESS_NETWORK_STATE, CHANGE_WIFI_MULTICAST_STATE for mDNS, CAMERA for QR;
   `SovereignSyncPlugin.kt`, `SharePlugin.kt`). Only re-run `cargo tauri android
   init` if it's stale — and re-apply those manifest/plugin patches if you do.

## 4. Gotchas (from CLAUDE.md + this work)

- **`sovereign-ai` default feature is `cuda`.** For anything that isn't a CUDA
  desktop build, pass `--no-default-features`. The `mobile` feature set already
  excludes cuda, cpal-voice, native-tls-email, and the embedded browser.
- **Mobile DB is `kv-mem`** (rocksdb gated off) → data is process-lifetime only.
  Persistent SQLite-backed SurrealDB on Android is a v0.0.6 item.
- **Native build deps:** `clang`, `cmake`, `libclang-dev` (bindgen for ggml /
  rocksdb / surreal). `protobuf-compiler` if prompted.
- Linker is `rust-lld` on Linux — watch `--as-needed` ordering if you hit link errors.
- `version` in `tauri.conf.json` (0.0.5) drives the Android `versionName`; bump
  `versionCode` per store upload.
- **Cargo.lock is gitignored** in this repo (don't try to commit it).

## 5. Known limitations shipping in v0.0.5 (don't "fix" — tracked for v0.0.6)

## Build completion log — WSL2 / Linux (2026-05-30)

The §3 starting tasks all completed; debug APK produced at
`crates/sovereign-app/gen/android/app/build/outputs/apk/universal/debug/app-universal-debug.apk`
(versionName **0.0.5**, 718 MB debug). A non-trivial chain of fixes was needed
beyond what §3 anticipated — none of these were verified on Windows because
the feature combinations only get exercised by the WSL2/Android pipeline.
Recorded here so the next ship cycle starts from a known state.

1. **Cargo workspace** — added `[patch.crates-io]` to redirect
   `curve25519-dalek = "4.1.3"` at Signal's fork
   (`https://github.com/signalapp/curve25519-dalek`, tag
   `signal-curve25519-4.1.3`). Without it, zkgroup pulls both the registry and
   forked copies of `curve25519-dalek` into the graph and explodes with
   `RistrettoPoint` cross-crate type-identity mismatches. The patch had never
   been applied because no build had ever compiled with `comms-signal`.

2. **`comms-signal` deferred from the `mobile` feature set.** Presage's
   `Manager::receive_messages()` returns a stream holding `ThreadRng`
   (`Rc<UnsafeCell<…>>` → `!Send`), which is incompatible with
   `CommunicationChannel`'s `Send + Sync` `#[async_trait]` bound and propagates
   non-Send futures across the orchestrator. A `LocalSet`-backed wrapper to
   bridge `!Send` presage to the Send-only outer trait is v0.0.6 work.
   `signal.rs` was simultaneously brought up to the presage 0.7.0 API
   (`OnNewIdentity`, 0-arg `receive_messages`, `Received` enum,
   `ContentBody::DataMessage { body, .. }` destructuring,
   `ServiceId::parse_from_service_id_string`, `Box::pin`-ing the stream) so
   the eventual re-enable starts from a working API surface.

3. **Embedded browser gated for Android/iOS.** `tauri::Webview` on mobile has
   neither `set_position`/`set_size` nor `show`/`hide` — those only exist on
   desktop where webviews can float inside a parent window. `mod browser`,
   `mod browser_pii`, the `tauri_commands::browser::{open,close,navigate,…}`
   entries, and the embedded-browser parts of `tauri_commands::pii` (form
   extraction, autofill) are now `#[cfg(not(any(target_os = "android",
   target_os = "ios")))]`. The `autofill_pii_record` fallback was widened to
   serve mobile too (returns an error explaining desktop-only).

4. **`cli_integration::test_cli_round_trip` gated behind `rocksdb`.** The test
   runs the binary as subprocesses sharing a temp DB path; mobile (no
   `rocksdb`) falls back to in-memory so each subprocess gets a fresh DB and
   the second invocation can't find the doc the first created. Now
   `#[cfg_attr(not(feature = "rocksdb"), ignore = "…")]`.

5. **Android Gradle build fixes:**
   - `gen/android/buildSrc/.../BuildTask.kt` had a hardcoded
     `C:\Program Files\nodejs\node` from the original Windows setup. Now picks
     `cargo tauri ...` on Linux/macOS, keeps the Windows path on Windows.
   - `SharePlugin.kt` / `SovereignSyncPlugin.kt` imported `app.tauri.plugin.PluginCall`
     which no longer exists in Tauri 2.11.2 — renamed to `Invoke`.
     `SharePlugin.onNewIntent` needed the `override` modifier (the base class
     declares `open fun onNewIntent`).

### Build command that worked

```bash
cargo tauri android build --debug --target aarch64 --features mobile -- --no-default-features
```

The `-- --no-default-features` passthrough is essential: tauri-cli has no
direct flag for it, and without it Tauri's android build pulls in the
package's desktop default (rocksdb) which can't cross-compile.
