Spatial Canvas & Versioning

Your documents live on an infinite canvas. Time runs left to right. Projects run top to bottom. Every edit is automatically remembered.

1
Tier 1: The Elevator Pitch
10-second read

Imagine your workspace as a living map. Documents on an infinite canvas. Time runs left→right. Projects run top→bottom. Owned content is rectangles; external is parallelograms. Every document remembers its complete edit history—automatically, invisibly. No save button. No lost work.

2
Tier 2: How It Works
1–3 minute read
The Two Axes

Horizontal: Timeline. Vertical: Intent Threads.

The canvas has two organizing dimensions:

  • Horizontal axis: Time. Past on the left, future on the right. A month-year ruler marks eras. Earlier edits live left; recent work drifts right.
  • Vertical axis: Intent Threads (projects). Each row is a lane. "Q1 Planning" lane stays in place. Documents within that lane can slide left/right in time, but stay in the same lane.
  • Layout invariant: A document's vertical position = its thread. Its horizontal position = when it was created or last touched.
Three Zoom Levels

Dots → Blocks → Full Cards

  • Dots (zoom < 0.3): Each document is a single colored dot. Color = intent thread. Useful for seeing structure at 10,000 feet.
  • Simplified (0.3–0.6): Small blocks with document title (truncated). Relationships shown as thin lines.
  • Full (≥ 0.6): Complete card: title, preview snippet, last-edited timestamp, relationships as curves, ownership badge.

Zoom transitions are smooth. No jank. Renderer culls off-screen cards automatically.

Sovereignty Halo: Owned vs External

Shape Encodes Ownership

  • Owned: Rounded rectangles. Sharp 90° corners softened to 8px radius.
  • External: Parallelograms (4° skew). Sheared edge signals "from outside".
  • Preattentive processing: Your eye catches shape before reading title. Ownership is instant.
  • Visual feedback: Hover any card: border glows, provenance label appears ("(owned)" or "(external)").
Relationships: A Web of Connections

Six Link Types, Drawn as Curves

  • References: Dashed blue line. "This mentions that."
  • DerivedFrom: Solid green arrow. "This version came from that version."
  • Continues: Dotted orange line. "This follows from that in narrative sequence."
  • Contradicts: Red X marker. "This conflicts with that."
  • Supports: Solid teal line. "This backs up that claim."
  • AttachedTo: Gray bracket. "This is metadata for that (e.g., a comment or tag)."

Curves flow across lanes, never overlapping (forced layout prevents spaghetti). Hover a card to highlight all its connections.

Built-In Versioning: Auto-Commit Engine

No Save Button. Just Save.

  • Edit threshold: After 50 keystrokes or actions, the system auto-commits with a snapshot of the document.
  • Time threshold: If 5 minutes pass since the last commit, commit now (even if only 10 edits).
  • On close: Low-activity docs auto-commit when you navigate away. No data loss.
  • Multi-device divergence: If you edit the same doc on two devices before syncing, you get a branch. Merge UI lets you reconcile.
  • Metadata: Each commit stores: document_id, parent_commit (forming a chain), message (auto-generated or user-written), timestamp, full snapshot.

The canvas shows time branches as subtle fork icons when hovering a doc.

ZoomLevel Enum (sovereign-canvas/src/zoom.rs)

The renderer uses a zoom level to determine what to draw. Three visibility tiers gate the complexity.

pub enum ZoomLevel { Dots, // zoom_factor < 0.3 Simplified, // zoom_factor >= 0.3 && < 0.6 Full, // zoom_factor >= 0.6 } impl ZoomLevel { pub fn from_factor(factor: f32) -> Self { match factor { f if f < 0.3 => ZoomLevel::Dots, f if f < 0.6 => ZoomLevel::Simplified, _ => ZoomLevel::Full, } } }

Auto-Commit Engine (sovereign-canvas/src/versioning.rs)

Every document has an associated commit scheduler. When thresholds are hit, a new commit is written to the DB.

ConstantValuePurpose EDIT_THRESHOLD50Number of edits before auto-commit. TIME_THRESHOLD_SECS300Seconds (5 min) before time-based auto-commit.
User types ↓ edit_count += 1, time_since_commit += 1 ↓ if edit_count >= 50 OR time_since_commit >= 300: ├─ Snapshot document state ├─ Create commit (parent=latest, message=auto-gen, snapshot=full) ├─ Write to DB └─ Reset counters ↓ On document close: ├─ If edit_count > 0: force commit └─ Cleanup timers

Commit Data Structure

Commits form a DAG (directed acyclic graph) where each commit points to its parent. Branches occur when the same doc is edited on two devices before sync.

pub struct Commit { pub id: String, // UUID pub document_id: String, pub parent_commit: Option, // Chain to prior version pub message: String, // Auto-gen or user-written pub timestamp: i64, pub snapshot: DocumentSnapshot, } pub struct DocumentSnapshot { pub document_id: String, pub title: String, pub content: String, pub metadata: Map, }

On merge, the system creates a "merge commit" with two parent_commit pointers (not yet implemented in MVP).

Rendering Architecture

The canvas uses Iced 0.14 and WGPU for GPU-accelerated rendering. The state is shared via Arc<Mutex> to allow concurrent edit + render loops.

pub struct CanvasState { pub zoom_factor: f32, pub pan_x: f32, pan_y: f32, // Scroll offset pub cards: Vec, pub lanes: Vec, pub selected_card: Option, } pub struct CanvasRenderer { state: Arc>, wgpu_device: Device, wgpu_queue: Queue, } // Render loop (decoupled from edit loop) impl CanvasRenderer { pub fn render(&self, frame: &Frame) { let state = self.state.lock(); // Draw lanes, cards, relationships } }

Canvas Commands

User interactions (mouse, keyboard, touch) are mapped to commands. The orchestrator executes them.
