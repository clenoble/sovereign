Reporting a Vulnerability

Responsible Disclosure

If you discover a security vulnerability, please report it responsibly. Do not post public issues, discussions, or pull requests about unpatched vulnerabilities.

How to Report

Primary channel: Email security@sovereignos.org

Encrypted reporting (PGP): Use our public key from SECURITY.md in the repository.

GitHub Security Advisories: Use GitHub's private disclosure feature for public repositories.

What to Include

  • Description of the vulnerability and its impact;
  • Affected version(s);
  • Steps to reproduce (if applicable);
  • Potential fix or mitigation (optional);
  • Your preferred timeline for disclosure (responsible disclosure window).

Response Timeline

Phase Timeline Description
Acknowledgment Within 48 hours We confirm receipt and begin triage.
Initial Assessment Within 7 days Severity classification and scope determination.
Fix Development Variable (see severity) Patch is developed and tested.
Public Disclosure Coordinated with reporter Patch is released; advisory is published.

MEDIUM 4.0–6.9 Within 30 days Denial of service, information disclosure

Scope

In Scope

  • All Sovereign OS crates: sovereign-core, sovereign-db, sovereign-crypto, sovereign-ai, sovereign-ui, sovereign-canvas, sovereign-skills, sovereign-p2p, sovereign-comms, sovereign-app;
  • Published releases (stable and pre-release);
  • Website (sovereignos.org) and documentation;
  • Official GitHub repositories.

Out of Scope

  • Third-party AI models (Qwen, Llama, Whisper, Piper) — report to model maintainers;
  • System dependencies (GTK, GStreamer, CUDA, ROCm, Python) — report to their maintainers;
  • Community-contributed skills (unless security issues in Sovereign OS skill framework);
  • Operational security of user devices (lost/stolen devices, malware, weak passwords);
  • Social engineering or phishing attacks;
  • Attacks on cloud providers or third-party services (report to them directly).

Hard Security Barriers

Constraint Description Enforcement
No data exfiltration Chat tools are read-only; AI cannot modify user data. Code (tool interface is read-only), not prompt.
Action gravity Destructive actions (delete, send) require user confirmation. Code (action gate enforces friction levels).
Encryption always-on User data at rest and in transit is encrypted by default. Architecture (no plaintext storage of sensitive data).
Injection detection Suspicious patterns in AI output are flagged to user. Code (injection module checks for attack patterns).
Prompt separation User input is not concatenated into system prompts. Code (ChatML format separates roles).

Prompt Injection Boundary

Sovereign OS uses the ChatML format to separate user input from system instructions. User messages are tagged with role boundaries that prevent prompt injection:

<|im_start|>system
You are a helpful assistant...
<|im_end|>

<|im_start|>user
[user input - cannot break system prompt]
<|im_end|>

<|im_start|>assistant
[AI response]
<|im_end|>

Even if user input contains "Ignore your instructions," it cannot affect the system prompt due to role boundaries.

Threat Model

Threat Actor Assumed Capability Primary Defense
Malicious Website Can serve malicious skill code Skill sandboxing, permission prompts, code review
Compromised Skill Can read user data if permission granted Granular permissions, audit log, user awareness
Stolen Device Physical access to encrypted storage Master key encryption, secure key derivation (Argon2)
Malicious Guardian Can block recovery (denial of service) Multiple guardians, offline recovery option
Network Attacker Can eavesdrop on P2P sync traffic End-to-end encryption, noise protocol
Compromised AI Model Can inject instructions into responses Injection detection, read-only tool interface

Disclosure Policy

Coordinated Disclosure Steps

  1. Reporter submits vulnerability via secure channel.
  2. Sovereign OS team acknowledges receipt within 48 hours.
  3. Team assesses severity and develops fix.
  4. Fix is tested on all supported versions.
  5. Advisory is drafted in private (CVE if applicable).
  6. Reporter and team agree on disclosure date (typically 30 days after patch release).
  7. Patch is released to all channels (crates.io, GitHub releases, package managers).
  8. Advisory is published with credit to reporter (if desired).

Safe Harbor

Researchers acting in good faith are protected under responsible disclosure. We will not pursue legal action against security researchers who:

  • Follow this policy and do not disclose vulnerabilities publicly before a fix is available;
  • Do not exploit vulnerabilities beyond the minimum necessary to demonstrate them;
  • Do not access, modify, or destroy user data;
  • Act with reasonable care to avoid disruption.

Bug Bounty

Sovereign OS currently does not offer a formal bug bounty program due to our volunteer, non-profit nature. However, we provide:

  • Recognition: Researchers who report vulnerabilities are credited in security advisories (if desired);
  • Gratitude: We deeply appreciate responsible research that improves security;
  • Future plans: As the project grows, we may establish a bounty program.
